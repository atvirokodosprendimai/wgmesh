# Specification: Issue #680

## Summary

Fix the spec-auto-approve workflow to use the GitHub App token consistently for all operations (approval, labeling, comments) instead of mixing GitHub App token with `GITHUB_TOKEN`, which causes approvals to be attributed to `github-actions[bot]` instead of the GitHub App actor (e.g., "wgmesh-ci[bot]").

## Context

The `.github/workflows/spec-auto-approve.yml` workflow validates and auto-approves Copilot spec PRs. Currently, it has inconsistent token usage:

- **Correct**: The `gh pr review --approve` command (line 180) uses `steps.app-token.outputs.token` (GitHub App token)
- **Incorrect**: The `gh pr edit --add-label approved-for-build` command (line 189) uses `GITHUB_TOKEN` via `ISSUE_WRITE_TOKEN`
- **Incorrect**: The `gh pr comment` for validation failures (line 264) uses `GITHUB_TOKEN`
- **Incorrect**: The scheduled scan job (line 299) creates a separate `issueGithub` instance using `GITHUB_TOKEN` for label operations

This mixed token usage means:
- The PR review approval appears from the GitHub App actor (e.g., "wgmesh-ci[bot]")
- The label addition and comments appear from `github-actions[bot]`

This breaks approval attribution consistency and creates unclear audit trails. The `bot-pr-review-merge.sh` script depends on consistent actor attribution to identify bot-authored PRs and auto-merge them. When the approval and label come from different actors, it can cause confusion in audit logs and break assumptions in downstream automation.

## Requirements

### Functional Requirements
- All spec PR approval operations (review, label, comment) must use the same authentication token
- The token must be the GitHub App token (`steps.app-token.outputs.token`), not `GITHUB_TOKEN`
- Both the event-driven job (triggered on PR events) and scheduled scan job (cron fallback) must use consistent tokens
- No changes to validation logic or workflow behavior beyond token substitution

### Non-Functional Requirements
- Minimal code changes (variable substitutions only)
- No changes to workflow permissions or GitHub App configuration
- Maintain existing error handling and retry logic
- Preserve all validation checks (spec file exists, required sections, no code changes, actionable classification)

### Technical Constraints
- Must use GitHub App token generated by `actions/create-github-app-token@v1` action
- App ID and private key are already configured in `vars.APP_ID` and `secrets.APP_PRIVATE_KEY`
- Existing permissions (`pull-requests: write`, `contents: write`, `issues: write`) are sufficient

## Acceptance Criteria

- All `gh` CLI commands in the `validate` job use `GH_TOKEN: ${{ steps.app-token.outputs.token }}` (via environment variable or inline)
- All GitHub REST API calls in the `scan` job use the GitHub App token-authenticated `github` context
- No references to `secrets.GITHUB_TOKEN` remain in the file
- No references to `ISSUE_WRITE_TOKEN` remain in the file
- The scheduled scan job uses a single GitHub client instance (no separate `issueGithub`)
- Create a test spec PR and verify all operations (approval, label, comments) appear from the GitHub App actor (e.g., "wgmesh-ci[bot]")
- Verify the `approved-for-build` label is added successfully
- Verify the Goose build workflow is triggered successfully after approval
- Confirm no changes to validation logic or approval criteria

## Out of scope

- Changes to GitHub App configuration or permissions
- Changes to validation logic or approval criteria
- Changes to the `bot-pr-review-merge.sh` script or other workflows
- Changes to the GitHub App ID or private key configuration
- Modifications to the scheduled scan timing or trigger conditions
- Changes to the metrics collection or artifact upload logic
- Modifications to the `goose-build.yml` workflow or its triggering mechanism
- Any changes to spec file validation rules or error messaging
