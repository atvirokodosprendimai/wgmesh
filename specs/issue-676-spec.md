# Specification: Issue #676

## Summary

Fix the spec-auto-approve workflow to use the GitHub App token consistently for all operations (approval, labeling, comments) instead of mixing GitHub App token with `GITHUB_TOKEN`, which causes approvals to be attributed to `github-actions[bot]` instead of the GitHub App actor (e.g., "wgmesh-ci[bot]").

## Context

The `.github/workflows/spec-auto-approve.yml` workflow validates and auto-approves Copilot spec PRs. Currently, it has inconsistent token usage:

### Event-driven Job (validate)
- **Line 168**: `GH_TOKEN: ${{ steps.app-token.outputs.token }}` - Correctly uses GitHub App token for checkout
- **Line 169**: `ISSUE_WRITE_TOKEN: ${{ secrets.GITHUB_TOKEN }}` - Creates variable pointing to generic GITHUB_TOKEN
- **Line 180**: `gh pr review` uses `GH_TOKEN: ${{ steps.app-token.outputs.token }}` - Correctly uses GitHub App token for approval
- **Line 189**: `GH_TOKEN="$ISSUE_WRITE_TOKEN" gh pr edit` - Uses GITHUB_TOKEN for label addition
- **Line 264**: `GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}` - Uses GITHUB_TOKEN for validation failure comments

### Scheduled Scan Job (scan)
- **Line 290-291**: Generates GitHub App token correctly
- **Line 299**: `ISSUE_WRITE_TOKEN: ${{ secrets.GITHUB_TOKEN }}` - Creates variable pointing to generic GITHUB_TOKEN
- **Line 301**: `github-token: ${{ steps.app-token.outputs.token }}` - Correctly uses GitHub App token for primary github client
- **Line 302-304**: Creates separate `issueGithub` instance using `GITHUB_TOKEN` for label operations

This mixed token usage means:
- The PR review approval appears from the GitHub App actor (e.g., "wgmesh-ci[bot]")
- The label addition appears from `github-actions[bot]`
- Validation failure comments appear from `github-actions[bot]`
- The scheduled scan uses two different actors for different operations

### Impact

1. **Confusing PR history**: Developers see mixed actors (GitHub App vs github-actions) for what should be a single automated operation
2. **Audit trail issues**: The approval timeline shows multiple actors, making it unclear which system performed the approval
3. **Downstream automation assumptions**: The `bot-pr-review-merge.sh` script's author normalization logic expects consistent bot attribution
4. **Trust problems**: Teams may question why approval and labeling come from different actors

## Requirements

### Functional Requirements
1. All spec PR approval operations (review, label, comment) must use the same authentication token
2. The token must be the GitHub App token (`steps.app-token.outputs.token`), not `GITHUB_TOKEN`
3. Both the event-driven job (triggered on PR events) and scheduled scan job (cron fallback) must use consistent tokens
4. No changes to validation logic or workflow behavior beyond token substitution

### Non-Functional Requirements
1. Minimal code changes (variable substitutions only)
2. No changes to workflow permissions or GitHub App configuration
3. Maintain existing error handling and retry logic
4. Preserve all validation checks (spec file exists, required sections, no code changes, actionable classification)

### Technical Constraints
1. Must use GitHub App token generated by `actions/create-github-app-token@v1` action
2. App ID and private key are already configured in `vars.APP_ID` and `secrets.APP_PRIVATE_KEY`
3. Existing permissions (`pull-requests: write`, `contents: write`, `issues: write`) are sufficient

## Acceptance Criteria

### Event-driven Job (validate)
1. All `gh` CLI commands use `GH_TOKEN: ${{ steps.app-token.outputs.token }}` (via environment variable)
2. No references to `secrets.GITHUB_TOKEN` remain in the validate job
3. No references to `ISSUE_WRITE_TOKEN` remain in the validate job
4. The `gh pr edit --add-label approved-for-build` command uses GitHub App token
5. The `gh pr comment` command for validation failures uses GitHub App token

### Scheduled Scan Job (scan)
1. All GitHub REST API calls use the GitHub App token-authenticated `github` context
2. No separate `issueGithub` instance is created
3. All operations (approval, labeling, workflow dispatch) use the same `github` client
4. No references to `secrets.GITHUB_TOKEN` remain in the scan job
5. No references to `ISSUE_WRITE_TOKEN` remain in the scan job

### Verification
1. Create a test spec PR and verify all operations (approval, label, comments) appear from the GitHub App actor (e.g., "wgmesh-ci[bot]")
2. Verify the `approved-for-build` label is added successfully
3. Verify the Goose build workflow is triggered successfully after approval
4. Confirm no changes to validation logic or approval criteria

## Out of Scope

1. Changes to GitHub App configuration or permissions
2. Changes to validation logic or approval criteria
3. Changes to the `bot-pr-review-merge.sh` script or other workflows
4. Changes to the GitHub App ID or private key configuration
5. Modifications to the scheduled scan timing or trigger conditions
6. Changes to the metrics collection or artifact upload logic
7. Modifications to the `goose-build.yml` workflow or its triggering mechanism
8. Any changes to spec file validation rules or error messaging
