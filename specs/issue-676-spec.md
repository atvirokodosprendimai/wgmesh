# Issue #676 Implementation Spec

## Summary

Fix the spec-auto-approve.yml workflow so that auto-approvals on spec PRs are attributed to the GitHub App identity itself (as a self-approval from the app), not as approvals from `github-actions[bot]`. Currently, the workflow approves PRs using a GitHub App token, but the approval actor appears incorrectly in the GitHub UI and audit logs.

## Context

The spec-auto-approve.yml workflow validates spec PRs and auto-approves them when they pass validation checks. This approval is intended to trigger the Goose implementation workflow.

Currently, the workflow uses `gh pr review` with a GitHub App token generated by `actions/create-github-app-token@v1`. The issue is that the approval appears as coming from `github-actions[bot]` instead of being attributed to the GitHub App itself as a self-approval.

The relevant code is in `.github/workflows/spec-auto-approve.yml`:

```yaml
- name: Auto-approve and trigger Goose
  if: steps.validate.outputs.valid == 'true'
  env:
    GH_TOKEN: ${{ steps.app-token.outputs.token }}
    ISSUE_WRITE_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

The problem is that `steps.app-token.outputs.token` is a GitHub App token, and when used with `gh pr review`, the approval actor shows as `github-actions[bot]` rather than the GitHub App itself.

## Requirements

### Functional Requirements

1. **FR-1**: Auto-approvals from spec-auto-approve.yml must appear as GitHub App self-approvals (from the app identified by `APP_ID`), not from `github-actions[bot]`

2. **FR-2**: The approval must still trigger the downstream Goose implementation workflow (goose-build.yml) via the `approved-for-build` label

3. **FR-3**: The approval comment body must remain unchanged (users see the same validation summary)

### Technical Requirements

1. **TR-1**: Use the GitHub App token for the `gh pr review` command such that the approval actor is the GitHub App itself

2. **TR-2**: Ensure the GitHub App has write permissions on pull requests (already configured in workflow permissions)

3. **TR-3**: Verify the fix by checking the approval actor in the PR review timeline after the workflow runs

### Non-Requirements

- Changing the validation logic or checks
- Modifying the scheduled scan job (which has a different approval flow)
- Changing the goose-build.yml workflow

## Acceptance Criteria

### AC-1: Approval Actor Identity
- **Given**: A spec PR passes all validation checks in spec-auto-approve.yml
- **When**: The workflow auto-approves the PR
- **Then**: The approval actor in the PR review timeline is the GitHub App (not `github-actions[bot]`)
- **Verification**: Check `gh pr view <pr_num> --json reviews --jq '.reviews[] | {author: .author.login, state: .state}'`

### AC-2: Label Application
- **Given**: A spec PR passes validation and is auto-approved
- **When**: The auto-approve step completes
- **Then**: The PR has the `approved-for-build` label applied
- **Verification**: `gh pr view <pr_num> --json labels --jq '.labels[].name'` contains `approved-for-build`

### AC-3: Goose Build Triggered
- **Given**: A spec PR is auto-approved and labeled
- **When**: The auto-approve step completes
- **Then**: The goose-build.yml workflow is triggered with correct issue_number and spec_pr_number inputs
- **Verification**: Check workflow runs in Actions tab for goose-build.yml

### AC-4: No Regression in Validation
- **Given**: A spec PR with validation failures (missing sections, code changes, etc.)
- **When**: The spec-auto-approve.yml workflow runs
- **Then**: The PR is NOT approved and a validation failure comment is posted
- **Verification**: PR has no approval and has a comment starting with "## Spec Validation Failed ❌"

### AC-5: Approval Comment Content
- **Given**: A spec PR passes validation and is auto-approved
- **When**: The auto-approve step completes
- **Then**: The approval review comment contains the validation summary (✅ Spec file exists, ✅ Required sections present, etc.)
- **Verification**: `gh pr view <pr_num> --json reviews --jq '.reviews[0].body'` contains "Auto-Approved ✅"

## Out of Scope

- Modifying the scheduled scan job (job `scan` in the same workflow) which has a separate approval mechanism using GitHub API calls
- Changing the approve-build.yml workflow which handles human approvals
- Modifying the bot-pr-review-merge.yml workflow
- Changing validation logic or checks
- Modifying the goose-build.yml workflow
- Changing how GitHub App tokens are generated
- Modifying PR review polling logic in bot-pr-review-merge.sh
