# Issue #539: Add Start(key []byte, fd int) and Stop() Methods for Android VPN API Integration

## Summary

Extend the `Daemon` interface with `Start(key []byte, fd int)` and `Stop()` methods to support Android VPN Service integration. The Android VPN API provides a pre-configured WireGuard tunnel device file descriptor (FD) and cryptographic key, requiring the daemon to initialize with these external resources instead of creating its own interface.

**Classification**: Feature Enhancement / Platform Integration
**Complexity**: Medium
**Priority**: High (enables Android platform support)

## Context

### Current Architecture

The `Daemon` currently manages the full WireGuard lifecycle:

1. **Interface Creation** (`pkg/daemon/helpers.go:createInterface`):
   - Linux: Uses `ip link add dev <name> type wireguard`
   - macOS: Uses `wireguard-go` userspace tunnel
   - Controlled via `setupWireGuard()` → `createInterface()` or `resetInterface()`

2. **Daemon Lifecycle** (`pkg/daemon/daemon.go`):
   ```go
   func NewDaemon(config *Config) (*Daemon, error)
   func (d *Daemon) Run() error
   func (d *Daemon) Shutdown()  // Currently just d.cancel()
   ```

3. **Configuration Flow**:
   - `initLocalNode()` loads/creates WireGuard keys
   - `setupWireGuard()` creates/configures the interface
   - `teardownWireGuard()` removes the interface on shutdown

### Android VPN API Constraints

Android's `VpnService` provides:
- A pre-configured tun/tap device FD (obtained via `VpnService.Builder::establish()`)
- The app must pass this FD to WireGuard instead of creating its own interface
- The VPN Service lifecycle controls when the tunnel is active
- Keys may be provided by the VPN system or generated by the app

### Integration Challenge

The current daemon assumes full control over interface lifecycle. For Android:
- Interface creation must be **skipped** (FD already exists)
- Keys must be **accepted as parameters** (may be externally managed)
- Shutdown must **preserve the FD** (VPN Service owns it)
- Daemon must be **startable/stoppable** independently from `Run()`

## Requirements

### 1. New Start Method Signature

```go
// Start initializes the daemon with an external WireGuard device (Android VPN mode).
// key: WireGuard private key (base64-encoded or raw bytes, spec clarification needed)
// fd: File descriptor of the pre-configured tun device (from VpnService.establish())
// Returns error if initialization fails, but does not block on peer operations.
func (d *Daemon) Start(key []byte, fd int) error
```

**Behavior**:
- Parse/validate the provided key (format TBD: raw bytes or base64 string)
- Configure the WireGuard interface using the FD instead of creating a new one
- Skip `createInterface()` and `resetInterface()` calls
- Initialize peer store, RPC server, and discovery layers
- Return immediately after initialization is complete
- **Must NOT delete or close the FD** on shutdown

### 2. New Stop Method

```go
// Stop gracefully shuts down the daemon without removing the WireGuard interface.
// Intended for Android VPN Service lifecycle management where the FD is owned externally.
func (d *Daemon) Stop() error
```

**Behavior**:
- Cancel all background goroutines (reconcile loop, health monitor, etc.)
- Stop discovery layers (DHT, LAN, gossip)
- Stop RPC server
- Flush peer state
- **Must NOT call `teardownWireGuard()` or delete the interface**
- Close the daemon's internal context

### 3. Modified Daemon Structure

**New fields** (if needed):
```go
type Daemon struct {
    // ... existing fields ...
    externalFD     int           // File descriptor for Android VPN mode
    isAndroidVPN   bool          // Flag to indicate external interface ownership
    externalKeySet bool          // Track if key was provided via Start()
}
```

**Configuration considerations**:
- `Config.InterfaceName` may be ignored or overridden by Android's naming
- `Config.WGPort` may need to be applied to the external FD differently
- Key validation must match the expected format (raw vs base64)

### 4. Key Format Handling

**Decision Point**: Specify whether `key []byte` is:
- **Option A**: Raw 32-byte private key (WireGuard standard)
- **Option B**: Base64-encoded string as bytes (wg display format)

**Recommended**: Option A (raw bytes) for consistency with Go crypto APIs. Conversion helpers:
```go
// Helper to convert base64 key to raw bytes if needed
base64KeyToRaw(base64Key string) ([]byte, error)
```

### 5. Interface Configuration with External FD

When `externalFD > 0`:
- Replace `wg set <iface>` commands with direct netlink/Socket operations
- OR: Determine if `wg` tool can operate on a specific FD (research needed)
- Set ListenPort, PrivateKey, and peers on the provided FD
- **Critical**: Ensure no `ip link delete` or similar destructive operations

### 6. Backward Compatibility

- Existing `NewDaemon()` + `Run()` flow must remain unchanged
- `Start()` is **optional** - daemon can still use traditional interface creation
- If `Start()` is called, `Shutdown()` becomes a no-op for interface cleanup
- `Run()` after `Start()` should be disallowed or documented as unsupported

## Acceptance Criteria

### Functional Requirements

1. **Start Method**:
   - [ ] Accepts a 32-byte raw WireGuard private key
   - [ ] Accepts a valid file descriptor (fd >= 0)
   - [ ] Configures WireGuard on the provided FD without creating a new interface
   - [ ] Initializes all daemon subsystems (peer store, discovery, RPC)
   - [ ] Returns error on invalid key or FD
   - [ ] Does **not** delete or close the FD on failure

2. **Stop Method**:
   - [ ] Stops all daemon goroutines (reconcile, health, discovery)
   - [ ] Flushes peer state cleanly
   - [ ] Does **not** call `teardownWireGuard()`
   - [ ] Does **not** close or delete the external FD
   - [ ] Returns nil on clean shutdown

3. **Integration Behavior**:
   - [ ] Traditional `NewDaemon()` + `Run()` unchanged (backward compatible)
   - [ ] `Start()` followed by `Stop()` leaves interface intact
   - [ ] `Start()` with invalid parameters fails fast with descriptive error
   - [ ] Multiple `Start()` calls are idempotent or return error (spec behavior)

4. **Platform-Specific Handling**:
   - [ ] `runtime.GOOS == "android"` detection used for conditional logic (if needed)
   - [ ] FD operations are portable (Linux/Android)
   - [ ] No hardcoded Android-specific APIs in core daemon code

### Testing Requirements

1. **Unit Tests** (`pkg/daemon/daemon_test.go`):
   - [ ] Test `Start()` with valid key and FD
   - [ ] Test `Start()` with invalid key (length, format)
   - [ ] Test `Start()` with invalid FD (< 0)
   - [ ] Test `Stop()` after `Start()` verifies interface still exists
   - [ ] Test backward compatibility: `NewDaemon()` + `Run()` unchanged
   - [ ] Test idempotence: calling `Start()` twice

2. **Integration Tests** (Android-specific, may be mock-based):
   - [ ] Mock FD creation and verify configuration
   - [ ] Verify peer operations work with external FD
   - [ ] Verify discovery layers start/stop correctly
   - [ ] Test graceful shutdown with active peers

3. **Manual Testing** (Android device):
   - [ ] Verify VPN Service integration with real `VpnService.Builder`
   - [ ] Test VPN connect/disconnect lifecycle
   - [ ] Verify tunnel data flow after `Start()`

## Out of Scope

The following are **explicitly excluded** from this specification:

1. **Android VPN Service Implementation**: No changes to `android/` or platform-specific VPN code (if it exists) - only daemon interface changes

2. **Key Generation/Rotation**: No changes to how keys are derived or stored. The `Start()` method receives a pre-existing key but does not manage key lifecycle

3. **Dynamic FD Switching**: No support for changing the FD mid-operation. Must `Stop()` and `Start()` with new FD

4. **Windows/iOS Support**: This spec is Linux/Android-specific. Other platforms are not considered

5. **Advanced FD Operations**: 
   - Pass-through of FD to child processes
   - FD duplication or inheritance
   - Raw socket operations beyond standard WireGuard configuration

6. **Configuration Format Changes**: No changes to `Config` struct or YAML/CLI parsing for Android-specific settings. Key and FD are runtime parameters only

7. **Permission/SELinux Handling**: Android permission model is out of scope; assume VPN Service grants necessary FD access

8. **Performance Optimization**: Initial implementation should focus on correctness. FD-specific optimizations (zero-copy, etc.) are deferred

## Open Questions

1. **Key Format**: Should `key []byte` be raw 32-byte values or base64 string bytes?
   - **Recommendation**: Raw bytes (convert in caller if needed)

2. **FD Ownership Verification**: Should `Start()` verify the FD is a valid tun device?
   - **Recommendation**: Basic validation (fd >= 0) only; trust VPN Service provides valid FD

3. **ListenPort Configuration**: How is `WGListenPort` applied to an external FD?
   - **Research Needed**: Can `wg` tool operate on an FD by path? Or use netlink?

4. **Error Recovery**: If `Start()` fails partway through, what cleanup is needed?
   - **Recommendation**: Log errors, return early, do not touch FD (caller owns lifecycle)

5. **Concurrent Operations**: Are `Start()` and `Stop()` thread-safe with existing daemon operations?
   - **Recommendation**: Use existing mutex/context patterns; document as not thread-safe for initial implementation

## Implementation Notes

1. **FD Handling**: Use `syscall.NewFileDescriptor` or similar to manage the FD safely. Ensure no garbage collection closes it unexpectedly.

2. **Key Validation**: Validate key length is exactly 32 bytes (WireGuard curve25519 private key size).

3. **Interface Detection**: When `externalFD > 0`, skip `interfaceExists()` check - the interface exists by definition.

4. **Graceful Shutdown**: Ensure all goroutines respect `d.cancel()` before `Stop()` returns. Add timeouts if necessary.

5. **Logging**: Add clear log messages distinguishing Android VPN mode from traditional mode:
   ```go
   log.Printf("Starting daemon in Android VPN mode (FD: %d)", fd)
   ```

6. **Testing Strategy**: Use interface mocking for the FD in unit tests. Consider a `MockFD` type that implements expected operations without requiring root privileges.
